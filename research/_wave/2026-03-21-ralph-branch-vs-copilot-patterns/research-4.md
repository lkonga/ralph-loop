## Findings

### Two Distinct Branch/Isolation Models

Copilot has **two separate systems** for branch management, used by different agent modes:

#### 1. Cloud Agent (Remote/Delegation) — `CopilotCloudGitOperationsManager`

**File**: `src/extension/chatSessions/vscode-node/copilotCloudGitOperationsManager.ts`

This handles the **"Delegate to cloud agent"** flow, where work is sent to GitHub's server-side agent.

**Branch Naming**:
- Pattern: `copilot/vscode-{base36_timestamp}-{random_4chars}`
- Example: `copilot/vscode-m4k9x2-a7b3`
- Generated by `generateRandomBranchName(repository, 'copilot')` (line 215)
- Uses `Date.now().toString(36)` for timestamp + `Math.random().toString(36).slice(2, 6)` for randomness
- **Collision avoidance**: Checks `refs/heads/{name}` up to 5 attempts; if all collide, falls back to just timestamp (`copilot/vscode-{base36_timestamp}`)

**Dirty-State Handling** (the `commitAndPushChanges()` flow, line 99):
1. Generates a random branch name
2. Creates the branch locally (`repository.createBranch(asyncBranch, true)` — the `true` means checkout)
3. Commits ALL changes with message `"Checkpoint from VS Code for cloud agent session"` using `commit(message, { all: true })`
4. Validates commit succeeded — checks HEAD matches new branch AND no remaining working tree or index changes
5. If auto-commit fails → opens an interactive terminal for the user with a 5-minute timeout (`handleInteractiveCommit`)
6. Pushes the new branch to remote with `set-upstream` (`push(remoteName, asyncBranch, true)`)
7. **Switches back** to the original base branch via `switchBackToBaseRef()`
8. On any error → **rollback**: checks out original branch via `rollbackToOriginalBranch()`

**Key detail**: The committed changes branch (`head_ref`) is passed to the remote agent API as a separate field from `base_ref`. The cloud agent uses `head_ref` as the starting point that includes the user's uncommitted work.

**Confirmation Dialog** (`buildConfirmation()`, line 1732):
- Before delegating, detects uncommitted changes via `detectedUncommittedChanges()` (checks `workingTreeChanges` + `indexChanges`)
- Also checks if the current branch exists on remote (`checkBaseBranchPresentOnRemote`)
- Presents buttons: "Commit Changes and Delegate", "Delegate" (without committing), "Authorize" (if needed), "Cancel"
- If branch isn't on remote: offers "Push Branch and Delegate"

**After agent finishes**: The cloud agent creates a **pull request** on GitHub. The user interacts with it through the sessions panel — they can view the PR, see file changes, and open in browser. PRs that are CLOSED or MERGED are filtered out of the sessions list.

#### 2. Background Agent (Local/CLI) — `ChatSessionWorktreeService`

**File**: `src/extension/chatSessions/vscode-node/chatSessionWorktreeServiceImpl.ts`

This handles the **local background agent** (agent mode sessions that work in worktrees).

**Branch Naming** (line 65):
- First tries `generateRandomBranchName()` from VS Code's git extension (returns a random name)
- Prepends user's `git.branchPrefix` config + `copilot/`
- Pattern: `{branchPrefix}copilot/{randomName}` (where `randomName` comes from VS Code's Git API)
- Fallback: `{branchPrefix}copilot/worktree-{ISO_timestamp}` (e.g., `copilot/worktree-2026-03-21T14-30-00`)

**Dirty-State Handling** (`FolderRepositoryManager`, line 274):
- Before creating a worktree, checks for uncommitted changes (`checkIfRepoHasUncommittedChanges`)
- Prompts user with 4 options:
  - **Copy Changes**: Copies uncommitted changes to the new worktree (originals stay)
  - **Move Changes**: Moves uncommitted changes to the worktree (source is cleaned)
  - **Skip Changes**: Creates worktree without migrating changes
  - **Cancel**: Aborts the operation
- Migration uses `gitService.migrateChanges()` with `deleteFromSource` flag based on move vs copy

**Worktree Properties** (interface in `chatSessionWorktreeService.ts`):
- Tracks: `baseCommit`, `branchName`, `baseBranchName`, `repositoryPath`, `worktreePath`
- V1: `autoCommit: boolean` — whether changes are auto-committed
- V2: `baseBranchName: string` + optional `pullRequestUrl`

**Applying Changes Back** (`applyWorktreeChanges`, line 155):
- V1 (legacy): Uses `migrateChanges()` (stash-like approach) from worktree to main repo
- V2: Generates a `diffBetweenPatch` and applies it via `gitService.applyPatch()`
- Also supports `mergeWorktreeChanges()`: checks out base branch → merges worktree branch
- And `updateWorktreeBranch()`: rebases worktree branch on top of base branch

### Remote Agent API Payload

**File**: `src/extension/chatSessions/vscode-node/copilotCloudSessionsProvider.ts` (line 2429)

The `RemoteAgentJobPayload` sent to GitHub's API includes:
```
{
  problem_statement: <truncated prompt + context>,
  event_type: 'visual_studio_code_remote_agent_tool_invoked',
  pull_request: {
    title: <extracted from prompt, max 20 chars + "...">,
    body_placeholder: "Cloud agent has begun work on **{title}**...",
    base_ref: <current branch or repo default>,
    head_ref: <optional: branch with committed dirty changes>,
    body_suffix: "Created from VS Code"
  }
}
```

### Base Branch Resolution

The `base_ref` sent to the cloud agent is determined by:
1. Check if current branch exists on remote
2. If yes → use current branch name
3. If no → fall back to repository's `default_branch` (from GitHub API)
4. For selected repos in multi-root workspaces → always fetch `default_branch` from GitHub API

## Patterns

| Aspect | Cloud Agent | Background Agent (Local) |
|--------|-------------|-------------------------|
| **Branch prefix** | `copilot/vscode-` | `{branchPrefix}copilot/` |
| **Uniqueness** | base36 timestamp + 4 random chars | VS Code's git `generateRandomBranchName()` or ISO timestamp |
| **Collision check** | Yes, up to 5 retries via `getRefs()` | No (relies on VS Code's API being unique) |
| **Dirty handling** | Auto-commit-all + push, or interactive terminal fallback | Prompt: copy/move/skip/cancel changes to worktree |
| **Isolation** | Creates temp branch, pushes, switches back | Creates git worktree (separate working directory) |
| **Switch back** | Always returns to original branch after push | N/A — worktree is a separate directory |
| **Rollback on error** | Yes, `checkout(baseRef)` on failure | Continues without isolation if worktree creation fails |
| **Result delivery** | Pull request on GitHub | Changes in worktree; apply via patch, merge, or migrate |
| **User interaction** | PR review/merge/discard via GitHub | Apply changes to main repo, merge branch, or discard worktree |

### Key Design Decisions

1. **Cloud agent never stashes** — it creates a temporary branch, auto-commits everything, pushes it, then switches back. The branch name includes `head_ref` in the API payload so the server knows where to find dirty state.

2. **Background agent uses worktrees for isolation** — no branch switching on the main repo. The user's main working directory is undisturbed.

3. **Branch names are intentionally opaque** — no semantic slug from the prompt. Cloud uses `copilot/vscode-{hash}`, background uses `copilot/{random}`. The PR title carries the semantic meaning.

4. **Graceful degradation** — if auto-commit fails, the cloud agent opens an interactive terminal for the user. If worktree creation fails, background agent falls back to working in the current workspace.

## Applicability

For a ralph-loop-style local agent:

1. **Branch naming**: The `copilot/vscode-{timestamp36}-{random4}` pattern is simple and effective. Collision checking via `getRefs()` is smart but rarely needed given timestamp uniqueness.

2. **Dirty-state strategy**: Two viable approaches:
   - **Cloud-style**: Auto-commit to temp branch → push → switch back. Good when you need server-side work.
   - **Worktree-style**: Create git worktree for true isolation. Best for local agents — user's workspace is never disturbed.

3. **Worktree > branch-switch for local agents**: The background agent's worktree approach is clearly superior for local use — no branch switching, no need to restore state, changes can be applied/merged/discarded independently.

4. **Change migration**: The copy/move/skip pattern for uncommitted changes is user-friendly and worth adopting. The patch-based `applyWorktreeChanges` (V2) is cleaner than stash-based migration (V1).

5. **PR as delivery mechanism**: For cloud agents, the PR is the handoff artifact. For local agents, the worktree branch can be merged directly or used to create a PR.

## Open Questions

1. **How does the cloud agent's server-side component create its own branch?** The VS Code extension only pushes a `head_ref` with dirty changes — the actual working branch for the agent's implementation is created server-side (not visible in this codebase). The PR's `headRefName` is stored in session metadata.

2. **What happens to the temporary `copilot/vscode-*` branches?** They're pushed to the remote but there's no cleanup code visible. Are they deleted after the cloud agent creates its own branch/PR?

3. **Worktree cleanup**: The code shows worktree creation and change application, but explicit worktree deletion/cleanup logic wasn't found in the examined files. How are stale worktrees handled?

4. **Interactive commit timeout**: The 5-minute timeout for manual commits in the cloud flow seems arbitrary. Is there telemetry on how often users need this fallback?

5. **Multi-repo support**: The cloud agent supports selecting from multiple repositories in multi-root workspaces, but worktree creation appears to use only the first/active repository. Is worktree creation supported across multiple repos?
