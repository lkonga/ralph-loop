# Final Report: ralph-loop VS Code Extension Status Bar Inconsistency

## Executive Summary
- The inconsistency is real and is explained by two overlapping defects, not one.
- In `ralph-loop`, terminal handlers update the status bar from `getStateSnapshot()` before `LoopOrchestrator.start()` resets `this.state` to `Idle` in `finally`, so snapshot-driven UI can stay on `running`/spinner while `ralph-loop.status` later reports `idle`. [via: aggregation-1.md#L9-L10 ← research-1.md#L8-L26] [via: aggregation-1.md#L9-L10 ← research-1.md#L44-L78] [via: aggregation-1.md#L9-L10 ← research-2.md#L16-L32] [via: aggregation-1.md#L9-L10 ← research-2.md#L33-L49] [via: aggregation-1.md#L9-L10 ← research-2.md#L50-L113] [via: aggregation-1.md#L9-L10 ← research-2.md#L91-L125] [via: aggregation-1.md#L9-L10 ← research-3.md#L180-L197] [via: aggregation-1.md#L9-L10 ← research-1.md#L27-L43]
- The literal fork status bar label `"processing"` is primarily explained by a second bug: Ralph-loop and `RequestLifecycleModel` are uncoordinated writers, so a later lifecycle `Processing` write can overwrite a correct Ralph `idle` push. [via: aggregation-2.md#L9-L9 ← research-5.md#L111-L132] [via: aggregation-2.md#L9-L9 ← research-5.md#L180-L194]
- This is why clicking the bar or running `ralph-loop.status` can show `idle` while the visible bar still says `processing`: the command reads live orchestrator state, but the UI depends on pushed snapshots/events that can lag or be overwritten. [via: aggregation-2.md#L18-L18 ← research-5.md#L111-L132] [via: aggregation-2.md#L18-L18 ← research-6.md#L9-L46]
- Normal terminal exits already race; abnormal exits are worse. `BranchEnforcementFailed`, `PrdValidationFailed`, uncaught exceptions, and resumed-session/auto-resume flows can miss a final Ralph idle refresh entirely. [via: aggregation-1.md#L11-L15 ← research-1.md#L27-L43] [via: aggregation-1.md#L11-L15 ← research-1.md#L44-L78] [via: aggregation-1.md#L11-L15 ← research-1.md#L79-L89] [via: aggregation-1.md#L11-L15 ← research-2.md#L16-L32] [via: aggregation-1.md#L11-L15 ← research-2.md#L50-L90] [via: aggregation-1.md#L11-L15 ← research-3.md#L46-L97] [via: aggregation-1.md#L11-L15 ← research-3.md#L109-L149] [via: aggregation-1.md#L11-L15 ← research-3.md#L180-L197] [via: aggregation-1.md#L11-L15 ← research-3.md#L241-L252]
- `fireStateChangeNotification` is not the faulty terminal-state emitter; it already pushes hardcoded terminal states. The real issues are stale Ralph snapshots and fork-side overwrite precedence. [via: aggregation-2.md#L11-L11 ← research-5.md#L9-L28] [via: aggregation-2.md#L11-L11 ← research-5.md#L29-L82]
- The highest-coverage fix is layered: make Ralph `idle` authoritative in the fork status bar, add a guaranteed post-`await orchestrator.start()` idle refresh (including auto-resume), and then make terminal snapshots truthful by resetting state earlier or emitting a final post-reset idle event. - Stop-latency and abort-unaware waits widen the stale-UI window but do not by themselves explain the exact `"processing"` label. They are hardening work, not the primary root cause. ## Consolidated Findings

### Core root causes
- **Late idle transition in Ralph-loop.** `LoopState.Idle` is assigned after terminal callbacks have already triggered snapshot-based UI updates, so terminal-event renders can publish stale `running` or spinner state even though later command reads are `idle`. [via: aggregation-1.md#L9-L9 ← research-1.md#L8-L26] [via: aggregation-1.md#L9-L9 ← research-1.md#L44-L78] [via: aggregation-1.md#L9-L9 ← research-2.md#L33-L49] [via: aggregation-1.md#L9-L9 ← research-2.md#L50-L113] [via: aggregation-1.md#L9-L9 ← research-3.md#L180-L197] [via: aggregation-2.md#L10-L10 ← research-4.md#L133-L171] [via: aggregation-2.md#L10-L10 ← research-6.md#L47-L102]
- **Competing writers in the fork status bar.** The visible `"processing"` text is best explained by fork-layer precedence problems: Ralph-loop and `RequestLifecycleModel` both write to the same bar, and a later lifecycle `Processing` write can replace a correct Ralph `idle` push. [via: aggregation-2.md#L9-L9 ← research-5.md#L111-L132] [via: aggregation-2.md#L9-L9 ← research-5.md#L180-L194]
- **Split state channels create the command/UI disconnect.** `fireStateChangeNotification` emits terminal `Idle` correctly, while Ralph status rendering reads snapshot/live state through a different path; meanwhile `ralph-loop.status` reads `this.state` directly. That split lets the command say `idle` while the status bar remains `processing`. [via: aggregation-1.md#L10-L10 ← research-2.md#L16-L32] [via: aggregation-1.md#L10-L10 ← research-2.md#L91-L125] [via: aggregation-1.md#L10-L10 ← research-1.md#L27-L43] [via: aggregation-2.md#L11-L11 ← research-5.md#L9-L28] [via: aggregation-2.md#L11-L11 ← research-5.md#L29-L82] [via: aggregation-2.md#L18-L18 ← research-5.md#L111-L132] [via: aggregation-2.md#L18-L18 ← research-6.md#L9-L46]

### Exit-path coverage and persistence
- **Normal terminal handlers still race.** `AllDone`, `MaxIterations`, `YieldRequested`, and `Stopped` call `updateStatusBar`, but they do so before the idle transition, so they can still paint stale running state. [via: aggregation-1.md#L11-L11 ← research-1.md#L27-L43] [via: aggregation-1.md#L11-L11 ← research-1.md#L44-L78] [via: aggregation-1.md#L11-L11 ← research-2.md#L16-L32] [via: aggregation-1.md#L11-L11 ← research-2.md#L50-L90]
- **Several failure paths skip a final Ralph refresh.** `BranchEnforcementFailed`, `PrdValidationFailed`, and uncaught `runLoop()` exceptions can fall through without a final Ralph idle update, which leaves stale UI visible indefinitely. [via: aggregation-1.md#L12-L12 ← research-1.md#L79-L89] [via: aggregation-1.md#L12-L12 ← research-3.md#L46-L97] [via: aggregation-1.md#L12-L12 ← research-3.md#L109-L149] [via: aggregation-1.md#L12-L12 ← research-3.md#L180-L197]
- **Resume flows have parity gaps.** The resumed-session callback only logs resumed events and does not update the Ralph status bar, so auto-resume can inherit the same stale-bar failure mode. [via: aggregation-1.md#L15-L15 ← research-3.md#L241-L252] [via: aggregation-2.md#L14-L14 ← research-6.md#L129-L135]
- **The current bar does not self-heal if idle never arrives.** `statusBar.ts` hides only when it receives an idle snapshot with no task id, so missing the final idle refresh leaves the stale bar visible rather than recovering on its own. [via: aggregation-1.md#L13-L13 ← research-1.md#L90-L102]

### Secondary contributing factors
- **Stop latency increases perceived inconsistency.** `stop()` is fire-and-forget, `delay()` ignores abort, and long execute paths can keep the loop alive after the user expects shutdown. This amplifies desync windows but does not fully explain the exact `"processing"` label. [via: aggregation-2.md#L12-L12 ← research-4.md#L25-L43] [via: aggregation-2.md#L12-L12 ← research-4.md#L67-L88] [via: aggregation-2.md#L12-L12 ← research-4.md#L107-L132]
- **The reports converge on a layered fix, not mutually exclusive fixes.** Caller-side final idle refresh gives the broadest coverage; earlier idle state or a final orchestrator event makes snapshots truthful; fork-side precedence prevents lifecycle `Processing` from clobbering Ralph `idle`. [via: aggregation-1.md#L14-L14 ← research-1.md#L139-L151] [via: aggregation-1.md#L14-L14 ← research-2.md#L126-L138] [via: aggregation-1.md#L14-L14 ← research-3.md#L213-L240] [via: aggregation-2.md#L14-L14 ← research-4.md#L174-L201] [via: aggregation-2.md#L14-L14 ← research-5.md#L192-L194] [via: aggregation-2.md#L14-L14 ← research-6.md#L129-L135]

## Pattern Catalog

### 1. Late terminal-state snapshots
**What happens:** terminal events read Ralph-loop state before `start()` finishes its `finally` reset, so snapshot-driven UI updates can never be fully trustworthy at shutdown. [via: aggregation-1.md#L18-L18 ← research-1.md#L44-L78] [via: aggregation-1.md#L18-L18 ← research-2.md#L50-L113] [via: aggregation-1.md#L18-L18 ← research-3.md#L180-L197] [via: aggregation-2.md#L17-L17 ← research-4.md#L133-L171] [via: aggregation-2.md#L17-L17 ← research-5.md#L66-L82] [via: aggregation-2.md#L17-L17 ← research-6.md#L62-L102]

**Implementation details:**
- Prefer moving `this.state = Idle` before terminal event handlers consume `getStateSnapshot()`.
- If that is too invasive, emit a final orchestrator-driven idle event after the reset.
- Keep a caller-side post-`await orchestrator.start()` idle refresh as a safety net because it also covers exceptions and early returns. [via: aggregation-1.md#L20-L20 ← research-1.md#L139-L143] [via: aggregation-1.md#L20-L20 ← research-2.md#L126-L138] [via: aggregation-1.md#L20-L20 ← research-3.md#L213-L232] [via: aggregation-2.md#L19-L19 ← research-4.md#L174-L201] [via: aggregation-2.md#L19-L19 ← research-5.md#L192-L194] [via: aggregation-2.md#L19-L19 ← research-6.md#L129-L135]

### 2. Competing status-bar writers without precedence
**What happens:** the fork status bar accepts both Ralph-loop updates and lifecycle updates, so a correct Ralph `idle` push can be overwritten by a later lifecycle `Processing` write. [via: aggregation-2.md#L9-L9 ← research-5.md#L111-L132] [via: aggregation-2.md#L13-L13 ← research-5.md#L121-L132] [via: aggregation-2.md#L13-L13 ← research-4.md#L133-L171] [via: aggregation-2.md#L13-L13 ← research-6.md#L62-L102]

**Implementation details:**
- Add merge/precedence logic so Ralph terminal `idle` is authoritative over lifecycle `processing` for the same surfaced task.
- Consider suppressing or scoping lifecycle `Processing` updates while Ralph-loop owns the bar.
- Only add reconciliation/polling if push+merge still leaves edge-case drift after precedence is enforced. [via: aggregation-2.md#L24-L24 ← research-5.md#L111-L132] [via: aggregation-2.md#L24-L24 ← research-5.md#L192-L194] [via: aggregation-2.md#L28-L28 ← research-5.md#L135-L145] [via: aggregation-2.md#L28-L28 ← research-5.md#L186-L193]

### 3. Missing final refreshes on non-happy exits and resume flows
**What happens:** some failure paths and resume flows do not produce a final Ralph idle update, so stale UI can persist forever. [via: aggregation-1.md#L21-L21 ← research-1.md#L79-L89] [via: aggregation-1.md#L21-L21 ← research-3.md#L46-L97] [via: aggregation-1.md#L21-L21 ← research-3.md#L109-L149] [via: aggregation-1.md#L21-L21 ← research-3.md#L241-L252]

**Implementation details:**
- Add a guaranteed caller-side idle refresh after `await orchestrator.start()` settles.
- Backfill explicit Ralph bar updates for `BranchEnforcementFailed`, `PrdValidationFailed`, and uncaught exception paths.
- Apply the same refresh path to resumed-session/auto-resume flows for parity. [via: aggregation-1.md#L26-L29 ← research-1.md#L139-L143] [via: aggregation-1.md#L26-L29 ← research-2.md#L126-L138] [via: aggregation-1.md#L26-L29 ← research-3.md#L213-L232] [via: aggregation-1.md#L27-L29 ← research-1.md#L79-L89] [via: aggregation-1.md#L27-L29 ← research-3.md#L46-L97] [via: aggregation-1.md#L27-L29 ← research-3.md#L109-L149] [via: aggregation-1.md#L27-L29 ← research-3.md#L233-L240] [via: aggregation-1.md#L29-L29 ← research-3.md#L241-L252] [via: aggregation-2.md#L26-L26 ← research-6.md#L103-L120] [via: aggregation-2.md#L26-L26 ← research-6.md#L133-L135]

### 4. Stop latency expands stale-UI windows
**What happens:** stop requests do not immediately wake all waits, which increases the time stale status can remain visible after user intent changes. [via: aggregation-2.md#L12-L12 ← research-4.md#L25-L43] [via: aggregation-2.md#L12-L12 ← research-4.md#L67-L88] [via: aggregation-2.md#L12-L12 ← research-4.md#L107-L132]

**Implementation details:**
- Make `delay()` abort-aware.
- Verify `executionStrategy.execute()` observes the abort signal promptly.
- Treat this as hardening after the primary status-bar ownership and idle-refresh fixes land. [via: aggregation-2.md#L27-L27 ← research-4.md#L67-L88] [via: aggregation-2.md#L27-L27 ← research-4.md#L121-L132] [via: aggregation-2.md#L27-L27 ← research-4.md#L191-L201]

## Recommended Plan
1. **Fix bar ownership first (no dependency).** Implement fork-side precedence so Ralph terminal `idle` wins over lifecycle `processing`. This directly addresses the literal user-visible `"processing"` label. [via: aggregation-2.md#L24-L24 ← research-5.md#L111-L132] [via: aggregation-2.md#L24-L24 ← research-5.md#L192-L194]
