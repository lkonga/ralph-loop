# Final Report: ralph-loop Feature Branch Enforcement Complexity / Net-Value Analysis

## Executive Summary (≤20 lines)

- Starting from an arbitrary non-protected branch such as `bisect/v0.39-lean` is currently risky because Phase 17 can silently adopt that branch, persist it to session state, and continue committing there instead of forcing alignment with the PRD-derived branch. [via: aggregation-1.md#L9-L9 ← research-1.md#L35-L45; aggregation-1.md#L9-L9 ← research-3.md#L31-L39; aggregation-1.md#L10-L10 ← research-1.md#L47-L110]
- The biggest hidden risk is continuous enforcement, not startup logic: the protected-branch gate runs once, but the intended per-commit backstop is effectively dead in orchestrator runs because `protectedBranches` is never passed into `atomicCommit`. [via: aggregation-2.md#L13-L13 ← research-5.md#L37-L38; aggregation-2.md#L13-L13 ← research-5.md#L61-L76; aggregation-2.md#L13-L13 ← research-6.md#L36-L41; aggregation-2.md#L13-L13 ← research-6.md#L122-L129]
- So the feature is good at bootstrapping the happy path, but weak at preventing or repairing wrong-branch drift after startup. [via: aggregation-2.md#L11-L11 ← research-6.md#L7-L32; aggregation-2.md#L11-L11 ← research-5.md#L44-L57; aggregation-2.md#L20-L20 ← research-5.md#L44-L57; aggregation-2.md#L20-L20 ← research-5.md#L61-L76; aggregation-2.md#L20-L20 ← research-6.md#L122-L129]

## Consolidated Findings (by category, deduplicated)

### Risk of starting from non-main / unexpected branches

- If startup happens on a non-protected branch that does not match the PRD-derived branch, the orchestrator currently adopts the current branch instead of blocking or asking. In practical terms, starting on `bisect/v0.39-lean` is risky whenever that branch is incidental rather than the intended work branch. [via: aggregation-1.md#L9-L9 ← research-1.md#L35-L45; aggregation-1.md#L9-L9 ← research-3.md#L31-L39; aggregation-1.md#L10-L10 ← research-1.md#L47-L110]
- Once adopted, the divergence becomes sticky: session persistence records the actual branch, `branchMismatch` is passive/runtime-only, and later commits keep targeting the adopted branch. [via: aggregation-1.md#L10-L10 ← research-1.md#L47-L110; aggregation-2.md#L12-L12 ← research-4.md#L7-L17; aggregation-2.md#L12-L12 ← research-4.md#L21-L45; aggregation-2.md#L12-L12 ← research-5.md#L38-L39; aggregation-2.md#L12-L12 ← research-6.md#L125-L129]
- The immediate failure mode is wrong-branch continuation more than catastrophic breakage; the more serious protected-branch exposure appears later if the branch drifts and the dormant commit-time guard never runs. [via: aggregation-2.md#L13-L13 ← research-5.md#L37-L38; aggregation-2.md#L13-L13 ← research-5.md#L61-L76; aggregation-2.md#L13-L13 ← research-6.md#L36-L41; aggregation-2.md#L13-L13 ← research-6.md#L122-L129; aggregation-1.md#L14-L14 ← research-3.md#L48-L55; aggregation-1.md#L14-L14 ← research-3.md#L84-L90; aggregation-1.md#L14-L14 ← research-1.md#L66-L78]
- Starts are materially safer when the current branch is protected and the system bootstraps the derived feature branch, when the current branch already is the expected branch, or when enforcement is explicitly disabled. [via: aggregation-2.md#L11-L11 ← research-6.md#L7-L32; aggregation-2.md#L11-L11 ← research-5.md#L44-L57]

### Net value vs. complexity

- The feature is not large or obviously over-engineered: one report characterizes it as roughly 109 source lines with substantial happy-path coverage and clear branch-enforcement events. [via: aggregation-2.md#L22-L22 ← research-5.md#L7-L24; aggregation-2.md#L22-L22 ← research-5.md#L44-L57; aggregation-2.md#L22-L22 ← research-5.md#L94-L107; aggregation-2.md#L22-L22 ← research-6.md#L7-L32; aggregation-2.md#L22-L22 ← research-6.md#L136-L141]
- The complexity that matters is semantic, not volumetric. The system has multiple branch truths—PRD-derived expected branch, stored session branch, runtime mismatch annotation, and commit-time guard intent—and they do not align into a single authority model. [via: aggregation-1.md#L19-L19 ← research-2.md#L79-L123; aggregation-1.md#L19-L19 ← research-1.md#L47-L64; aggregation-2.md#L12-L12 ← research-4.md#L7-L17; aggregation-2.md#L12-L12 ← research-4.md#L21-L45; aggregation-2.md#L21-L21 ← research-4.md#L38-L45; aggregation-2.md#L21-L21 ← research-4.md#L88-L94; aggregation-2.md#L21-L21 ← research-6.md#L125-L129; aggregation-2.md#L21-L21 ← research-6.md#L143-L148]
- Tests create false reassurance because helper-level or happy-path behavior is stronger than the riskiest production paths: silent branch adoption, mid-loop drift, resume checkout failure, title mutation, worktrees, dirty startup states, and non-standard default branches are under-tested or untested. [via: aggregation-1.md#L13-L13 ← research-1.md#L112-L121; aggregation-1.md#L13-L13 ← research-2.md#L124-L154; aggregation-1.md#L13-L13 ← research-3.md#L40-L66; aggregation-1.md#L13-L13 ← research-3.md#L70-L72; aggregation-2.md#L16-L16 ← research-4.md#L57-L72; aggregation-2.md#L16-L16 ← research-5.md#L18-L24; aggregation-2.md#L16-L16 ← research-6.md#L59-L60; aggregation-2.md#L16-L16 ← research-6.md#L85-L96; aggregation-2.md#L16-L16 ← research-6.md#L113-L118]
- Verdict: Phase 17 is a net positive as startup workflow assistance and branch ergonomics, but not yet as a robust branch-integrity guarantee. No material contradictions appear across the six research reports; they mostly converge on that scope boundary. ## Pattern Catalog (with implementation details and source refs)

### 1. Startup-only enforcement

The design invests in a clean startup decision tree and safe terminal failures, but enforcement is concentrated at loop entry. After startup, branch safety depends on nothing changing underneath the loop. [via: aggregation-2.md#L20-L20 ← research-5.md#L44-L57; aggregation-2.md#L20-L20 ← research-5.md#L61-L76; aggregation-2.md#L20-L20 ← research-6.md#L122-L129]

### 2. Fail-open adoption of arbitrary non-protected branches

Unexpected non-protected branches are silently adopted instead of blocked, warned, or explicitly confirmed. This optimizes continuity, but it normalizes wrong-branch execution and makes incidental branches look intentional. [via: aggregation-1.md#L9-L10 ← research-1.md#L35-L45; aggregation-1.md#L10-L10 ← research-1.md#L47-L110; aggregation-1.md#L18-L18 ← research-1.md#L35-L45; aggregation-1.md#L18-L18 ← research-3.md#L31-L39]

### 3. Dual branch authority

Resume logic treats stored `branchName` as truth, while startup logic treats the PRD-derived branch as truth; `branchMismatch` can detect divergence, but it is runtime-only and UI-only. The result is mechanically correct detection without coherent authority or reconciliation. [via: aggregation-2.md#L12-L12 ← research-4.md#L7-L17; aggregation-2.md#L12-L12 ← research-4.md#L21-L45; aggregation-2.md#L21-L21 ← research-4.md#L38-L45; aggregation-2.md#L21-L21 ← research-4.md#L88-L94; aggregation-2.md#L21-L21 ← research-6.md#L125-L129; aggregation-2.md#L21-L21 ← research-6.md#L143-L148]

### 4. Lossy PRD-title slugging

Branch derivation is deterministic, but it is also lossy: ASCII-only normalization, shared fallback buckets such as `ralph/prd`, raw truncation, and a missing-H1 crash path all weaken branch identity. That makes title mutation, duplicate titles, and untitled PRDs practical sources of orphan or collision branches. [via: aggregation-1.md#L11-L12 ← research-2.md#L45-L77; aggregation-1.md#L11-L11 ← research-2.md#L79-L123; aggregation-1.md#L11-L11 ← research-2.md#L146-L154; aggregation-2.md#L15-L15 ← research-5.md#L32-L32; aggregation-2.md#L15-L15 ← research-5.md#L40-L42; aggregation-2.md#L15-L15 ← research-6.md#L43-L50; aggregation-2.md#L15-L15 ← research-6.md#L98-L111; aggregation-2.md#L15-L15 ← research-6.md#L131-L132]

### 5. Coverage illusion

The test suite is strongest where the system is least controversial: startup logic, happy paths, and isolated helpers. The real-world gaps live in orchestration, resume failure handling, and adversarial repo states, so the system looks safer in tests than it is in production. [via: aggregation-1.md#L13-L13 ← research-1.md#L112-L121; aggregation-1.md#L13-L13 ← research-2.md#L124-L154; aggregation-1.md#L13-L13 ← research-3.md#L40-L66; aggregation-1.md#L13-L13 ← research-3.md#L70-L72; aggregation-2.md#L16-L16 ← research-4.md#L57-L72; aggregation-2.md#L16-L16 ← research-5.md#L18-L24; aggregation-2.md#L16-L16 ← research-6.md#L59-L60; aggregation-2.md#L16-L16 ← research-6.md#L85-L96; aggregation-2.md#L16-L16 ← research-6.md#L113-L118]

## Recommended Plan (ordered, with dependencies)

1. **Activate the dormant per-commit guard immediately.** Wire `protectedBranches` into both orchestrator `atomicCommit` call sites. This is the highest-confidence, lowest-effort safety repair and does not require redesigning the rest of the feature. [via: aggregation-1.md#L27-L27 ← research-3.md#L31-L39; aggregation-1.md#L27-L27 ← research-3.md#L84-L90; aggregation-2.md#L29-L29 ← research-5.md#L61-L76; aggregation-2.md#L29-L29 ← research-6.md#L36-L41; aggregation-2.md#L29-L29 ← research-6.md#L152-L160]
2. **Define the policy for unexpected non-protected branch starts.** Decide whether the default should be `block`, `warn`, or `confirm adopt`; document that policy; then make startup behavior explicit instead of silently fail-open. This step is the direct answer to the `bisect/v0.39-lean` risk. [via: aggregation-1.md#L26-L26 ← research-1.md#L35-L45; aggregation-1.md#L26-L26 ← research-1.md#L160-L169]
3. **Unify branch authority and persistence.** Persist both `expectedBranch` and `actualBranch`, consume mismatch state in the orchestrator instead of only in UI, and make failed Resume checkout blocking. This depends on step 2 because the persisted model should reflect the chosen startup policy. [via: aggregation-1.md#L28-L28 ← research-1.md#L61-L78; aggregation-1.md#L28-L28 ← research-1.md#L165-L167; aggregation-2.md#L30-L30 ← research-4.md#L38-L53; aggregation-2.md#L30-L30 ← research-4.md#L108-L110; aggregation-2.md#L30-L30 ← research-6.md#L154-L160]
4. **Harden branch naming after authority is clear.** Add Unicode-safe handling, missing-H1 protection, and branch disambiguation or warnings for duplicate / mutated / untitled PRD titles. This is easier once step 3 determines which branch identity fields are durable. [via: aggregation-1.md#L29-L29 ← research-2.md#L45-L77; aggregation-1.md#L29-L29 ← research-2.md#L79-L123; aggregation-1.md#L29-L29 ← research-2.md#L133-L154; aggregation-2.md#L32-L32 ← research-6.md#L43-L50; aggregation-2.md#L32-L32 ← research-6.md#L98-L111; aggregation-2.md#L32-L32 ← research-6.md#L156-L156]
5. **Backfill adversarial coverage as each repair lands.** Add regression tests for mid-loop drift, Resume checkout failure, detached HEAD, non-standard default branches, worktrees, dirty startup states, and concurrent fresh starts. This should happen incrementally during steps 1-4, not only afterward. [via: aggregation-2.md#L33-L33 ← research-4.md#L68-L72; aggregation-2.md#L33-L33 ← research-6.md#L59-L60; aggregation-2.md#L33-L33 ← research-6.md#L95-L96; aggregation-2.md#L33-L33 ← research-6.md#L113-L118]
6. **Instrument and measure before tightening further.** Once the above fixes land, measure how often collisions, worktree issues, and branch drift really occur, and whether any additional continuous checks have meaningful cost.
