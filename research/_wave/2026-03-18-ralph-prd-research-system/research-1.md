# Research Report: PRD.md Structure, Task Format, and Spec Pointers

**Question**: What is the structure and task format of PRD.md, and how are phases, task numbering, checkboxes, and `→ Spec:` line-range pointers encoded?

**Date**: 2026-03-18

---

## Findings

### 1. Overall PRD Structure

The PRD (`PRD.md`) is a flat markdown document organized into **numbered phases** using `## Phase N` or `## Phase N —` headers. Each phase contains:
- A blockquote (`>`) preamble describing the phase's origin, research source, and mandatory rules (TDD, verification commands).
- Optional subsections using `### Subsection Name` headers to group related tasks.
- Task entries as markdown checkbox list items.

**Phase numbering** is sequential (1–11 currently), with natural language titles: `## Phase 2 — Autopilot Patterns`, `## Phase 9 — Adaptive Intelligence`. Subsections within phases use `### ` headers (e.g., `### 9a — Context & Knowledge Intelligence`).

### 2. Task Format

Each task is a markdown checkbox line:

```
- [ ] **Task NN — Short Name**: Full description... → Spec: `path/to/file.md` LNN-LNN
- [x] **Task NN — Short Name**: Description (completed)
```

**Key elements**:
- **Checkbox**: `- [ ]` (pending) or `- [x]` (complete). Parsed by regexes in `src/prd.ts`:
  - `CHECKBOX_UNCHECKED = /^(\s*)-\s*\[\s*\]\s+(.+)$/`
  - `CHECKBOX_CHECKED = /^(\s*)-\s*\[x\]\s+(.+)$/i`
- **Bold task ID**: `**Task NN — Name**` prefix. The `parseTaskId()` function extracts the bold content as the task's human-readable identifier via `/^\*\*([^*]+)\*\*/`.
- **Sequential `taskId`**: Auto-generated by `parsePrd()` as `Task-001`, `Task-002`, etc. (zero-padded to 3 digits, based on parse order, NOT the `NN` in the description).
- **Numeric `id`**: A 0-based integer index assigned during parsing.

### 3. Task Annotations (DSL)

Tasks support inline annotations parsed by `parsePrd()`:

| Annotation | Regex | Effect |
|---|---|---|
| `[DECOMPOSED]` | string check `line.includes('[DECOMPOSED]')` | Task is skipped (non-actionable) |
| `[CHECKPOINT]` | string check `line.includes('[CHECKPOINT]')` | Sets `task.checkpoint = true`; orchestrator pauses for human review instead of executing |
| `[AGENT:name]` | `/\[AGENT:(\w+)\]/g` | Sets `task.agent` to the agent name; rejected if multiple annotations on same task |
| `[NO_DIFF]` | `/\[NO_DIFF\]/i` | Sets `task.noDiff = true`; skips diff validation for documentation tasks |
| `depends: task-id` | `/depends:\s*([\w,\s-]+)/i` | Sets `task.dependsOn` array for DAG-based execution ordering |

Phase-level `[AGENT:name]` on `### ` headers is inherited by all tasks in that phase unless overridden at the task level.

### 4. `→ Spec:` Line-Range Pointers

The `→ Spec:` annotation links a PRD task to a specific line range in a spec/research file:

```
→ Spec: `research/14-phase9-refined-tasks.md` L15-L36
```

**Parsing** in `src/prompt.ts` via `extractSpecReference()`:
```typescript
const match = taskDescription.match(/→\s*Spec:\s*`?([^`\s]+)`?\s+L(\d+)-L(\d+)/);
```
Returns `{ filePath, startLine, endLine }`.

**Usage**: `buildSpecContextLine(workspaceRoot, taskDescription)` reads the referenced spec file, extracts its YAML frontmatter, and generates a one-liner context injection: `[Spec context: Phase 9 | principles: configurable, composable | verify: tsc+vitest]`. This is injected into the prompt so the agent knows spec metadata without reading the full file.

The prompt also includes a **SPEC REFERENCE GATE** section instructing the executing agent: "If the task description references a spec file, you MUST read that file at the specified line range BEFORE writing any code."

### 5. Phase-Level Conventions

Each phase blockquote (`>`) typically includes:
- **Source attribution**: Where patterns were extracted from.
- **TDD rule**: "TDD is MANDATORY" (unconditional from Phase 5 onward).
- **Completion checklist**: `(1) npx tsc --noEmit, (2) npx vitest run, (3) append to progress.txt, (4) mark [x], (5) git commit`.

These are human-readable conventions — the parser (`parsePrd()`) does NOT extract them. The `buildPrompt()` function injects equivalent machine-readable sections (TDD GATE, SEARCH-BEFORE-IMPLEMENT GATE, SPEC REFERENCE GATE) into every prompt unconditionally.

### 6. The `Task` Interface (src/types.ts)

```typescript
interface Task {
  readonly id: number;           // 0-based parse index
  readonly taskId: string;       // "Task-001" format
  readonly description: string;  // Full text after checkbox, annotations stripped
  status: TaskStatus;            // Pending | InProgress | Complete | ...
  readonly lineNumber: number;   // 1-based line number in PRD.md
  readonly dependsOn?: string[]; // Task IDs this depends on
  readonly checkpoint?: boolean; // [CHECKPOINT] annotation
  readonly agent?: string;       // [AGENT:name] annotation
  readonly readOnly?: boolean;   // Derived from agent name (explore/research)
  readonly noDiff?: boolean;     // [NO_DIFF] or auto-detected documentation task
}
```

### 7. Task Picking Logic

- `pickNextTask(snapshot)` → delegates to `pickReadyTasks(snapshot, 1)[0]`
- `pickReadyTasks(snapshot, maxTasks)` → returns pending tasks whose `dependsOn` are all complete. Parallel safety: if any task in a batch uses a write agent, falls back to sequential (returns only first task).
- Documentation tasks are auto-detected via `isDocumentationTask()` using regex patterns for doc-related keywords.

### 8. PRD Validation

`validatePrd(snapshot)` checks:
1. Non-empty task list
2. No duplicate pending task descriptions (case-insensitive)
3. No dangling dependency references (warns)
4. No circular dependencies (DFS cycle detection)

---

## Patterns

1. **Flat-file DSL**: PRD.md is a self-contained, human-readable DSL. No external schema — structure is encoded in markdown conventions and parsed by regex.
2. **Annotation-in-description**: All metadata (`[CHECKPOINT]`, `[AGENT:name]`, `depends:`, `[NO_DIFF]`, `→ Spec:`) is embedded directly in the task description text, then stripped during parsing. This keeps the PRD editable by hand.
3. **Two-tier task IDs**: Human-readable IDs (from bold text, e.g., `Task 57 — Context Budget Awareness`) coexist with machine-assigned sequential IDs (`Task-001`). The `dependsOn` field uses the human-readable parsed ID.
4. **Spec pointers as lightweight references**: `→ Spec:` provides a Tier-2 task pattern — brief PRD description + full spec in a referenced file with line ranges. Keeps PRD concise while enabling deep specs.
5. **Progressive phase evolution**: Early phases (1-3) have inline task descriptions. Later phases (9+) use `→ Spec:` pointers. Phase preambles grow more structured (adding TDD rules, commit steps).
6. **Frontmatter sealing**: Spec files referenced by `→ Spec:` pointers use YAML frontmatter as sealed metadata (phase, tasks, principles, verification). `buildPrompt()` reads this for context injection.

---

## Applicability

- **For wave-orchestrator PRD generation**: The generator must output tasks in exact PRD format: `- [ ] **Task NN — Name**: Description` with optional `→ Spec: path LNN-LNN`. Must auto-detect next task number from existing PRD.
- **For context grounding**: `pickReadyTasks()` and `parsePrd()` are the key functions to understand task flow and dependency resolution.
- **For spec generation**: Spec files need YAML frontmatter with `type: spec`, `tasks: [NN, ...]`, `phase: N`, `verification`, and `principles` fields for `buildSpecContextLine()` to work.
- **For PRD write protection (Task 77)**: `validatePrdEdit()` must allow checkbox toggles, `[DECOMPOSED]` prefix, and `depends:` annotations while rejecting description text changes.

## Open Questions

1. **Task numbering gap risk**: PRD uses two numbering systems — the in-description `Task NN` (human-authored, can have gaps) and the auto-generated `Task-NNN` (sequential). Dependencies use the human-authored ID via `parseTaskId()`. If a task is removed or renumbered, dependencies may break silently (only caught by `validatePrd()`'s dangling reference check).
2. **Spec pointer prevalence**: Only Phase 9 tasks use `→ Spec:` pointers extensively. Phases 1-8 use fully inline descriptions. Is this intentional (phased adoption) or should older phases be retrofitted?
3. **Phase header detection**: `parsePrd()` matches any `#{1,6}` header for phase agent annotation, but does NOT extract phase numbers. Phase number detection is only in the spec generator agent (Task 26 description), not in `prd.ts`. This could cause issues if phase-level metadata is needed at runtime.
4. **`→ Spec:` line ranges are fragile**: If the spec file is edited, the line numbers in PRD task descriptions become stale. No automated validation exists for this.
