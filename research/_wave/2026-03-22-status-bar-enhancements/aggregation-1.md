## Aggregation Report 1

### Source Reports

- `research-1.md` finds that the shared token JSON stores full raw CAPI and optional OAuth tokens, but recommends using in-memory services for runtime UI and hashing raw token values before showing any compact fingerprint. [source: research-1.md#L23-L39] [source: research-1.md#L54-L83] [source: research-1.md#L84-L176]
- `research-2.md` maps the OAuth session flow, shows that `anyGitHubSession?.accessToken` is synchronously available from `IAuthenticationService`, and recommends wiring status-bar refresh to `onDidAuthenticationChange`. [source: research-2.md#L42-L97] [source: research-2.md#L98-L172]
- `research-3.md` traces `X-Initiator` and `X-Interaction-Id` header creation, notes that BYOK strips both headers before forwarding, and recommends injecting `IInteractionService` into `forkStatusBar.ts` to surface `interactionId.slice(-4)` during existing renders. [source: research-3.md#L8-L56] [source: research-3.md#L57-L164]

### Deduplicated Findings

- The shared token file is an exposure surface, not the preferred live data source for the status bar; the fresher and safer runtime sources are the in-memory DI services already maintained by the extension. [source: research-1.md#L23-L39] [source: research-1.md#L177-L190] [source: research-2.md#L136-L172]
- The raw CAPI token is intentionally hidden from `IAuthenticationService.copilotToken`; to fingerprint it for display, read the raw token from `ICopilotTokenStore.copilotToken.token` or `authService.getCopilotToken()`, then hash before truncating. [source: research-1.md#L56-L72] [source: research-1.md#L84-L165]
- The raw OAuth token is synchronously reachable through `authService.anyGitHubSession?.accessToken`; however, the merged recommendation is to treat it like the CAPI token and display a hashed suffix rather than the raw last four characters. [source: research-2.md#L57-L108] [source: research-2.md#L136-L172] [source: research-1.md#L125-L133]
- `createSha256Hash()` is the safest shared utility for both token fingerprints, with the UI showing only the last four hexadecimal characters of the digest when a compact hint is needed. [source: research-1.md#L86-L124] [source: research-1.md#L125-L165]
- Contradiction resolved: `research-2.md` treats raw OAuth last-4 display as acceptable for the status bar, while `research-1.md` argues raw token suffixes should never be shown; the lower-risk, more consistent policy is to hash both tokens before display. [source: research-2.md#L136-L163] [source: research-1.md#L125-L133]
- Token-related UI can refresh from existing events instead of polling: `onDidAuthenticationChange` covers auth state updates, and `tokenStore.onDidStoreUpdate` remains available if CAPI-specific refresh granularity is needed. [source: research-1.md#L166-L176] [source: research-2.md#L75-L86] [source: research-2.md#L164-L172]
- `X-Initiator` is derived from `userInitiatedRequest` inside `chatMLFetcher.ts`, not owned by the status bar; billing and classification logic can change it before header emission, and BYOK endpoints can strip it entirely. [source: research-3.md#L10-L56]
- `X-Interaction-Id` comes from `IInteractionService.interactionId`, is regenerated by `startInteraction()` on first-turn non-subagent requests, and is injected into both WebSocket and HTTP request paths. [source: research-3.md#L59-L85]
- The lowest-risk status-bar enhancement is to inject `IInteractionService` into `ForkStatusBarContribution` and append `interactionId.slice(-4)` in `_mergeAndRender()`, relying on existing request-lifecycle renders instead of adding a new platform event. [source: research-3.md#L95-L118] [source: research-3.md#L121-L164]

### Cross-Report Patterns

- Existing dependency-injected, in-memory services are the intended data plane for the status bar: OAuth via `IAuthenticationService`, CAPI via `ICopilotTokenStore`, and interaction IDs via `IInteractionService`. [source: research-1.md#L54-L83] [source: research-2.md#L57-L108] [source: research-3.md#L121-L164]
- The reports consistently favor minimal plumbing over new infrastructure: reuse existing events and render cycles instead of polling the shared file or expanding platform interfaces unless a real gap appears. [source: research-1.md#L166-L190] [source: research-2.md#L164-L172] [source: research-3.md#L121-L158]
- Compact observability beats full disclosure: use truncated, human-readable hints for debugging, but avoid exposing full runtime secrets or identifiers when a short fingerprint or suffix is sufficient. [source: research-1.md#L84-L165] [source: research-3.md#L86-L164]

### Gaps

- None of the reports benchmarks async hashing in the hot status-bar render path or decides whether token fingerprints should be cached per token value.
- None of the reports specifies the final UX format for text vs tooltip, collision handling if two token hashes share a suffix, or whether secure-display/screen-share modes should suppress identifiers.
- None of the reports provides end-to-end validation for sign-in, sign-out, CAPI rotation, or BYOK vs GitHub endpoint behavior after the status-bar change.
- `research-3.md` traces where `X-Initiator` is computed, but does not identify the cleanest already-exposed status-bar model field for showing the current initiator label without duplicating classification logic.
- The requested source paths under `ralph-loop` were absent, so this aggregation had to resolve the matching wave reports from `vscode-copilot-chat`; mirroring or indexing cross-repo wave artifacts would reduce future ambiguity.
